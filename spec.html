1. Overview

Blitzcrank is now a narrow TypeScript tool for Seerr/Jellyfin media-server operations. It receives Seerr issue webhooks, runs scheduled Markdown automations, owns service credentials and gateway state, and delegates all agentic reasoning to Pi through Pi RPC mode.

Pi owns the model/provider configuration, skills, durable agent sessions, tool loop, project TypeScript tools, and conversation history. Blitzcrank owns webhook ingress, SQLite issue state, Seerr final comments/resolution, and passing configured service credentials to Pi runs.

Current scope: Seerr issue workflows and scheduled automations only. Pi owns provider integrations, skills, sessions, and tool execution.

2. Features

Seerr Issue Gateway

Accepts Seerr webhooks, deduplicates events, locks per issue, runs Pi, posts final comments, and resolves issues when Pi returns RESOLVE_ISSUE: yes.

Pi Agent Runtime

Blitzcrank starts pi --mode rpc with persistent sessions, project skills, and a project extension that registers media-server tools.

Typed Service Tools

Pi calls seerr_request, jellyfin_request, sonarr_request, radarr_request, and sabnzbd_request; TypeScript tools call configured services directly and enforce service-relative paths.

Markdown Automations

Scheduled Markdown tasks run through Pi without durable Pi sessions and use live service state as source of truth.

Gateway Persistence

SQLite stores only Seerr issue gateway state. Pi JSONL sessions store agent conversation history.

Session Search

Pi sessions are searchable through thread_history_search for repeated issue context.

4. Workflows

4.1 Seerr Issue Workflow

  1. Seerr sends an issue webhook to seerr.webhook_path.
  2. Blitzcrank validates the optional webhook secret, classifies the event, and ignores unknown/duplicate/bot-authored events.
  3. The harness locks on the Seerr issue id and loads/updates the SQLite issue thread.
  4. Blitzcrank builds a compact issue prompt from the latest payload, prior events, and prior runs stored in SQLite.
  5. Pi runs in the persistent session for issue:<id>, loads .pi/skills/seerr-issue-solver/SKILL.md, and calls typed tools.
  6. Pi returns RESOLVE_ISSUE: yes/no followed by the public Seerr comment.
  7. Blitzcrank validates, signs, posts the final comment, and resolves the issue only for RESOLVE_ISSUE: yes.

4.2 Automation Workflow

Automations are Markdown files under automations/*.md with front matter: name, description, and schedule. Currently @hourly is supported. Each automation runs through Pi with a stable session named from automation:<name>.

The current bundled automation, hourly-stale-import-handler, uses Sonarr/Radarr queue and manual-import APIs to import safe stale downloads or remove clearly rejected stale queue items. It reports only concrete new imports, removals, or new manual-review blockers.

Pi sees a small typed tool surface registered by .pi/extensions/blitzcrank-tools.ts. The extension calls configured services directly using environment passed to the Pi process.

6. Pi Runtime

Blitzcrank uses Pi as the only agent runtime. Provider setup, API keys, OAuth, default models, skill loading, context management, and tool-loop behavior belong to Pi.

Blitzcrank controls only which Pi binary to call, working directory, persistent session location, per-task model override, and process environment.

[pi]
command = "pi"
cwd = "."
sessions_dir = "pi-sessions"

[pi.models]
default = "anthropic/claude-sonnet-4-5:high"
seerr = ""
automation = ""

Thinking/reasoning is configured inline in Pi's model string, for example anthropic/claude-sonnet-4-5:high.

8. Persistence & State

SQLite

SQLite stores only Blitzcrank gateway state:

  • issue_threads — Seerr issue lifecycle and status.
  • issue_thread_events — webhook event records and dedupe keys.
  • issue_runs — Pi run metadata, posted status, attribution, errors, and completion reason.

Old generic agent/Discord tables were removed. Pi owns agent conversation history.

Pi sessions

Pi sessions are stored under pi.sessions_dir for non-automation issue runs. Blitzcrank creates one durable session per Seerr issue using sanitized thread ids such as issue-42.jsonl. Automation runs are invoked without a durable Pi session.

History search

thread_history_search searches Pi session files.

9. Safety Model

  • Small tool surface: Pi only sees service request tools and session search.
  • Credential isolation: API keys are passed as process environment to Pi tools, not placed in prompts or tool arguments.
  • Relative paths only: service tools reject full URLs and credential-looking path parameters.
  • Mutation friction: non-GET requests require narrow_mutation and a concrete safety reason.
  • Harness-owned Seerr lifecycle: Pi cannot post final comments or resolve issues directly.
  • Evidence before action: skills require read-only service checks and validation after mutations.
  • Privacy: final comments must avoid raw JSON, internal URLs, service credentials, unrelated user data, and implementation details.

10. Deployment Notes

Run locally:

cp config.example.toml blitzcrank.toml
go run ./cmd/blitzcrank

Useful checks:

go test ./...
go build ./cmd/blitzcrank
nix build

Provider credentials are configured through Pi's normal environment/settings, for example ANTHROPIC_API_KEY, OPENAI_API_KEY, or Pi auth storage. Blitzcrank should not own provider setup.

Kagi web tools require KAGI_API_KEY in the service environment.
